**分支色不写入持久数据，默认作为运行时派生状态存在；渲染时通过原生 DOM 设置 CSS variable；当树结构变化、拖拽跨分支、折叠展开、导入旧文件时，重新计算分支归属。**

也就是说：

* `.naotu` 文件不强制迁移
* `branchColor` 默认不存进节点数据
* `branchIndex / branchColor` 放在运行时 projection 层
* DOM 渲染时使用 `style.setProperty`
* 拖拽跨分支后自动继承新分支颜色
* 折叠/展开后不改变颜色，但需要在重新投影或局部渲染时保持 branch meta

# 三、branchIndex / branchColor 存放位置

这里需要改得更明确。

我建议不要写入原始节点数据，而是放在 **ProjectionNode** 或一个运行时 Map 中。

## 推荐方案：放在 ProjectionNode

如果你当前已有布局投影层，推荐这样：

```ts
type SourceNode = {
  id: string;
  type: "text" | "notebook";
  parentId?: string;
  children: string[];
  title: string;
};

type ProjectionNode = {
  id: string;
  source: SourceNode;

  x: number;
  y: number;
  width: number;
  height: number;

  depth: number;
  parentId?: string;
  children: string[];

  branchIndex: number | null;
  branchColor: string | null;
};
```

这样逻辑很清楚：

* `SourceNode` 是数据源，负责保存用户内容
* `ProjectionNode` 是运行时视图模型，负责布局、坐标、分支色、可见性
* `branchColor` 是渲染属性，不污染 `.naotu` 数据

## 替代方案：使用运行时 Map

如果你暂时没有明确的 ProjectionNode，可以用：

```ts
type BranchMeta = {
  branchIndex: number | null;
  branchColor: string | null;
};

const branchMetaByNodeId = new Map<string, BranchMeta>();
```

渲染时：

```ts
const meta = branchMetaByNodeId.get(node.id);

if (meta?.branchColor) {
  nodeEl.style.setProperty("--branch-color", meta.branchColor);
}
```

但从长期维护看，我更建议放进 ProjectionNode。因为颜色、坐标、可见性、折叠后的投影，本质上都属于“视图派生状态”。

# 五、已有 `.naotu` 文件的迁移策略

这里应该补充，而且建议采用 **零迁移策略**。

因为分支色默认是运行时派生的，不需要写进 `.naotu` 文件。

## 推荐策略

### 旧文件加载

旧文件没有分支色字段：

```json
{
  "nodes": [...]
}
```

加载后：

1. 读取原始树结构
2. 找到中心节点
3. 按中心节点一级子节点顺序分配分支色
4. 递归给每个子树生成 branch meta
5. 渲染时使用运行时 branch meta

不修改原文件。

### 保存时

默认仍然不保存 `branchColor`：

```json
{
  "nodes": [...]
}
```

这样不会污染旧文件格式，也不会引入迁移复杂度。

### 以后如果支持用户手动锁定分支色

再引入可选字段：

```ts
type SourceNode = {
  id: string;
  type: "text" | "notebook";
  children: string[];

  branchColorOverride?: string;
};
```

只有用户手动设置颜色时才保存。

也就是说：

> 自动分支色不保存；用户自定义分支色才保存。

这是最稳的迁移策略。

# 六、拖拽节点跨分支时颜色如何变化

建议规则：

> 节点被拖拽到新的一级分支下后，整个子树立即继承新分支颜色。

例如：

```txt
中心
├─ A 分支：橙色
│  └─ x
└─ B 分支：蓝色
```

如果用户把 `x` 从 A 分支拖到 B 分支下：

```txt
中心
├─ A 分支：橙色
└─ B 分支：蓝色
   └─ x
```

那么 `x` 和它的所有后代都变成蓝色。

## 实现触发点

拖拽完成后：

```ts
function onNodeDrop(nodeId: string, newParentId: string, newIndex: number) {
  moveNode(nodeId, newParentId, newIndex);

  rebuildProjection();
  renderMindmap();
}
```

如果你后面要优化性能，可以改成局部更新。但第一版直接重建 projection 是最稳的。

## 注意

如果拖拽的是中心节点的一级子节点，并改变了一级分支顺序，则可能导致多个一级分支颜色重排。

例如：

```txt
中心
├─ A 橙色
├─ B 蓝色
├─ C 绿色
```

拖动 C 到第一位后：

```txt
中心
├─ C 橙色
├─ A 蓝色
├─ B 绿色
```

这会让颜色跟着“顺序”变。
这符合自动配色逻辑，但用户可能会感觉颜色跳变。

所以可以分两期：

## 第一版

颜色根据一级分支顺序自动分配。
拖拽一级分支重排时，颜色会跟着顺序变化。

## 第二版

支持“分支色锁定”。
一级分支一旦创建，自动获得一个 `branchColorOverride`，以后重排也不变。

我建议第一版先做自动顺序配色，等用户反馈后再做锁定。

# 七、折叠/展开时分支色的即时更新触发点

折叠/展开本身不改变节点的父子结构，所以理论上不需要重新计算分支色。

但它会改变：

* 可见节点集合
* 可见连线集合
* layout projection
* SVG path

因此折叠/展开后应该重新生成 projection，branch meta 顺便保持一致。

推荐触发流程：

```ts
function onToggleCollapse(nodeId: string) {
  toggleCollapsed(nodeId);

  rebuildProjection();
  renderMindmap();
}
```

其中：

```ts
function rebuildProjection() {
  const visibleTree = buildVisibleTree();
  const branchMeta = assignBranchMeta(visibleTree);
  const layout = computeLayout(visibleTree, branchMeta);

  return layout;
}
```

注意：

* 折叠节点的隐藏子孙虽然不渲染，但它们的数据还在
* 再展开时，子孙重新进入 projection
* 它们重新继承所属一级分支颜色

所以折叠/展开不会导致颜色丢失。

# 八、更新后的分支色计算逻辑

更适合文档中的版本如下。

```ts
type BranchPaletteItem = {
  color: string;
  soft: string;
  border: string;
};

const BRANCH_PALETTE: BranchPaletteItem[] = [
  {
    color: "var(--mindmap-branch-1)",
    soft: "var(--mindmap-branch-1-soft)",
    border: "var(--mindmap-branch-1-border)",
  },
  {
    color: "var(--mindmap-branch-2)",
    soft: "var(--mindmap-branch-2-soft)",
    border: "var(--mindmap-branch-2-border)",
  },
  {
    color: "var(--mindmap-branch-3)",
    soft: "var(--mindmap-branch-3-soft)",
    border: "var(--mindmap-branch-3-border)",
  },
];

function getBranchPaletteItem(index: number): BranchPaletteItem {
  return BRANCH_PALETTE[index % BRANCH_PALETTE.length];
}
```

生成 projection 时：

```ts
function assignBranchMetaToProjection(
  rootId: string,
  projectionById: Map<string, ProjectionNode>
) {
  const root = projectionById.get(rootId);
  if (!root) return;

  root.branchIndex = null;
  root.branchColor = null;

  root.children.forEach((childId, index) => {
    const palette = getBranchPaletteItem(index);
    assignBranchToSubtree(childId, index, palette, projectionById);
  });
}

function assignBranchToSubtree(
  nodeId: string,
  branchIndex: number,
  palette: BranchPaletteItem,
  projectionById: Map<string, ProjectionNode>
) {
  const node = projectionById.get(nodeId);
  if (!node) return;

  node.branchIndex = branchIndex;
  node.branchColor = palette.color;
  node.branchColorSoft = palette.soft;
  node.branchColorBorder = palette.border;

  for (const childId of node.children) {
    assignBranchToSubtree(childId, branchIndex, palette, projectionById);
  }
}
```

`ProjectionNode` 应为：

```ts
type ProjectionNode = {
  id: string;
  sourceId: string;

  type: "text" | "notebook";

  x: number;
  y: number;
  width: number;
  height: number;

  depth: number;
  parentId?: string;
  children: string[];

  isRoot: boolean;
  isCollapsed: boolean;
  isVisible: boolean;

  branchIndex: number | null;
  branchColor: string | null;
  branchColorSoft?: string | null;
  branchColorBorder?: string | null;
};
```

# 九、DOM 渲染函数示例

```ts
function applyBranchVars(el: HTMLElement | SVGElement, node: ProjectionNode) {
  if (!node.branchColor) return;

  el.style.setProperty("--branch-color", node.branchColor);

  if (node.branchColorSoft) {
    el.style.setProperty("--branch-color-soft", node.branchColorSoft);
  }

  if (node.branchColorBorder) {
    el.style.setProperty("--branch-color-border", node.branchColorBorder);
  }
}
```

节点渲染：

```ts
function renderNode(node: ProjectionNode): HTMLElement {
  const nodeEl = document.createElement("div");

  nodeEl.addClass("mindmap-node");
  nodeEl.addClass(node.type === "notebook" ? "notebook-node" : "text-node");

  if (node.isRoot) {
    nodeEl.addClass("root-node");
  }

  if (node.children.length > 0) {
    nodeEl.addClass("is-internal");
  } else {
    nodeEl.addClass("is-leaf");
  }

  applyBranchVars(nodeEl, node);

  nodeEl.style.left = `${node.x}px`;
  nodeEl.style.top = `${node.y}px`;

  return nodeEl;
}
```

连线渲染：

```ts
function renderLink(link: ProjectionLink): SVGPathElement {

  }

  if (link.branchColorBorder) {
    pathEl.style.setProperty("--branch-color-border", link.branchColorBorder);
```

# 十一、最终修正版原则

可以把文档里的原则改为这一版：

```md
脑图 UI 色彩系统以 branch color 为核心，而不是 warning/error/success 状态色。

1. 画布背景跟随 Obsidian 主题。
2. Text Node 与 Notebook Node 是用户可选择的两类节点。
3. 从中心节点出发的每个一级分支分配一个 branch color。
4. 该一级分支下所有后代节点和连线继承同一 branch color。
5. branch color 默认不写入 .naotu 文件，而是在运行时 projection 阶段计算。
6. 旧 .naotu 文件无需迁移，加载后按当前树结构自动生成 branch color。
7. 拖拽节点跨分支后，节点及其子树继承新分支颜色。
8. 折叠/展开不改变分支归属，但会触发 projection/layout/render 更新。
9. 蓝色 focus color 只表示 selected / focused / editing，不表示分支或节点类型。
10. 为兼容 Obsidian 的 Electron 环境，第一版不依赖 color-mix()，而是使用预定义的 color / soft / border 三套变量。
```

这样改完之后，整套方案会更接近真实可实现版本，也更适合你的 Obsidian 插件。
